## Findings — нестыковки и трудности

### F1. `contradict` слеп к fact↔reality

`fact_store(action='contradict')` вернул 0 противоречий. Однако факт 163 утверждал, что в MEMORY.md есть указатель на flight-search — а его там не было. Это противоречие «факт ↔ реальность», не «факт ↔ факт». Инструмент сравнивает только тексты фактов попарно и не верифицирует их против live state (файлы, cron, config).

**Главное загрязнение памяти — именно такого рода:** не взаимоисключающие утверждения, а snapshot'ы, устаревшие или неверные с момента записи.

### F2. `retrieval_count` мёртв

У всех 87 фактов `retrieval_count = 0`. Нельзя отличить активно используемый факт от мёртвого по числу обращений. Пришлось решать «canonical vs duplicate» по семантике вручную. Если бы retrieval работал, можно было бы автоматически: выше count → canonical, ниже → кандидат на удаление.

### F3. Серая зона «дубликат vs отдельный факт»

Flight-search guardrail факты 160/161/162/163 — не точные копии, а итеративные черновики. Каждый следующий чуть уточнял предыдущий. Без diff-инструмента внутри fact_store пришлось читать все четыре и собирать лучший вручную. С DeepSeek-кластером (31/34/155/156/159) та же проблема сложнее — каждый факт несёт разный аспект, и автоматический merge потеряет нюансы.

### F4. Volatile state как durable fact — антипаттерн без детекции

Факт 82 (snapshot cron-задач на дату) и факт 81 (версия skill v1.1.1) — runtime state, захардкоженный как fact. Они не противоречат ничему в fact_store, но протухли через неделю. Фильтра «это volatile?» нет — пришлось опознавать по смыслу.

### F5. Trust ≠ canonical

Среди группы дубликатов trust мог быть равным или выше у худшего варианта. Trust отражает «полезность при использовании», а не «каноничность». Это разные метрики.

### F6. Категория skill `.archive/` не вычищается

Stale cron ID `62e7a25f4e15` остался в `~/.hermes/skills/.archive/daily-knowledge-distillation/`. Активный skill пропатчен, но `.archive/` — это категория внутри структуры навыков (old/unused skills), не буквальный «архивный склад». Cleanup одного skill не означает cleanup всей цепочки зависимых категорий.

### F7. Процесс не масштабируется

87 фактов → полный ручной просмотр, группировка, 11 решений. На 200+ это долго и error-prone. Нет встроенного dedup/merge flow — только ручной search + judgment.

## Proposals

### P1. Verification pass после `contradict`

После `contradict` добавить шаг: для фактов, утверждающих состояние внешних объектов (файлы, cron, config), делать spot-check.

Скриптом: grep фактов по ключевым словам (`exists`, `enabled`, `runs`, `contains`, `current`, `latest`) и сверить с `stat`/`cronjob list`/`read_file`.

Приоритет: **высокий** — ловит главный класс загрязнения.

### P2. Volatile tag

При `fact_store add` использовать `tags="volatile"` для снапшотов/runtime state.

Гигиена: `tags=volatile` + возраст > N дней → кандидат на re-verify или remove.

Не ломает текущий формат — просто использует уже существующее поле `tags`.

Приоритет: **высокий** — простой, решает F4.

### P3. Canonical marker

Вместо trust как proxy на «главный факт в группе» — добавить тег `tags="canonical"` или отдельное поле.

Dedup-процесс: canonical остаётся, остальные с пересечением entities → кандидаты на merge/remove.

Приоритет: **средний** — упрощает F3/F5, но требует дисциплины при add.

### P4. Retrieval count — либо починить, либо убрать из логики

Если провайдер не чинит `retrieval_count` — обнулить ожидания и убрать из audit-логики.

Альтернатива: логировать свои обращения к фактам через обязательный `fact_feedback` после каждого `probe`/`search`/`reason` — это даст относительный signal.

Приоритет: **средний** — workaround, не архитектурное исправление.

### P5. Категория `.archive/` — включать в scope cleanup

При patch/update skill'а проверять `.archive/` категорию на stale ссылки.

Либо явно документировать: `.archive/` — intentional snapshot, не чистится автоматически.

Приоритет: **низкий** — мелочь, но предотвращает F6.

### P6. Batch audit prep-script

Скрипт, который groups факты по entity overlap, показывает кластеры > 1 факта, помечает candidates для ручного review.

Не автомерж — только препарация для human-in-the-loop.

Приоритет: **средний** — решает F7 для масштаба > 100 фактов.

## Risks / pitfalls

- `tags="volatile"` / `tags="canonical"` — convention, не enforcement. Забыли тег → факт не попадёт в фильтр.
- Verification pass по ключевым словам не поймёт неявные утверждения («backup runs nightly» → нет ключевого слова `enabled`).
- Batch prep-script может создать ложные кластеры (факты про один и тот же entity, но разные аспекты).
